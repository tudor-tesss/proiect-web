1. Introduction

1.1 Purpose & Product Scope

The software submitted to your viewing is called I Dislike It!, a web platform dedicated to the online community of people that wish to share their opinions about certain topics. The platform is designed to be used by anyone that has an account on it, and it is meant to be used by people that want to share their opinions about various topics. These topics are divided into categories, which may be created by the user and which should have various rating criteria. The user can then create a topic of discussion under a certain category, where they will rate the topic and provide a description of their opinion. Other users may also rate the topic and provide their own opinions.

1.2 Intended audience

This product is intended for anyone that wishes to share their opinions about various topics.

2. Overall description

2.1 Product perspective

I Dislike It! is a web platform that allows users to share their opinions. Its' specifications can be observed below.

2.2 Product functions

The product allows users to perform the following actions:

  • Sign up
  • Sign in
  • View all categories
  • View all posts in a category
  • View a post and all of its' replies
  • Create a category
  • Create a post
  • View statistics and export them
  • Dark mode for light sensitive eyes

2.3 User classes and characteristics

There are three main user types for the I Dislike It! application:

  1. Unregistered users: users that have not created an account on the platform. They do not have access to any pages other than the Sign in and Sign up pages.
  2. Registered users: users who have created an account on the platform. They can login, browse categories, browse posts, create categories and create new posts.

2.4 Operating environment

I Dislike It! is a web application that can be accessed from any device that has a web browser installed, and functions on any browser such as Chrome, Firefox, Safari, Opera, etc. (some display features may only be available in Chromium based browsers - i.e. the custom scroll bar).

2.5 Design and implementation constraints

The website has been designed and developed using HTML, CSS, and JavaScript for frontend development, while the backend is built using Java, with a custom-built library for routing and database management. The database is a PostgreSQL database, which is hosted on the same server as the backend.

  • Ensuring cross-browser compatibility
  • Adhering to web accessibility guidelines
  • Maintaining a consistent look and feel across the website
  • Optimizing the website for performance and load times

3. System Features

3.1 Account management

Users can create an account and login to the website. To avoid generic security issues such as password phishing and stealing, we have opted to use a passwordless authentication system, based on email gates. This means that the user, upon entering their email address on the sign in page, will receive an email with a code. Then, they will be prompted to enter the code in the website and we will verify that the code is correct. If it is, the user will be logged in.

3.2 Landing page (categories overview)

The landing page, or the categories overview page, is a page that displays all the categories that are available on the website. The categories are displayed in a grid with their name. The user can click on a category to access all of the posts in that category.

3.3 Category overview

Upon entering this page, users will be greeted with a list of all the posts in the category. The posts are displayed in a single column, and contain the title, the author, the rating criteria and the ratings given by the author, and the description of the post. The user can click on a post to view it in more detail.

3.4 Post overview

Upon accessing this page, users will be greeted with a post, which contains the title, the author, the rating criteria and the ratings given by the author, and the description of the post. The user can also see all the replies to the post, along with the authors and ratings of the new replies. The user may also reply to the post and express their own opinion.

3.5 Create category

Registered users can add new categories, which will be displayed on the landing page. The category name must be unique. The category must also contain between one and five rating criteria.

3.6 Add post

While viewing a category, registered users can add new posts to the category. The post must contain a title, a description, and a rating for each of the rating criteria.

3.7 Add reply

While viewing a post, registered users can add new replies to the post. The reply must contain a description, and a rating for each of the rating criteria.

3.8 Generate statistics

Registered users can generate statistics for a category or a post. The statistics will be available in CSV, DocBook or PDF format.

4. External Interface Requirements

4.1 User interfaces

The I Dislike It! website features a user-friendly interface that is easy to navigate and control. The colour palette is consistent throughout the website and generally follows a 60-30-10 rule, and was chosen to be easy on the eyes and to provide a comforting experience to the user. The use of calming colours such as green helps convey a sense of trust and safety. The website also features a custom scroll bar, which is only available in Chromium based browsers, for now. All of the buttons and links are clearly visible and easy to click on, and the website is responsive and adapts to different screen sizes. Here are some of the key features of the user interface:

  • Clear navigation menus and buttons
  • Responsive design that adapts to different screen sizes
  • Readable fonts and colors
  • Pleasant colour palette

4.2 Application Programming Interfaces (APIs)

The I Dislike It! may use external APIs, such as SendGrid, to send emails to users.

4.3 Hardware interfaces

As a web-based application, I Dislike It! does not have any hardware requirements. However, in order to access the website, users must have a device with a web browser installed, and an internet connection.

4.4 Software interfaces

I Dislike It! is built using HTML, CSS, and JavaScript for the frontend, and a server-side programming language (Java) and a database management system (PostgreSQL) for the backend. The platform is designed to work with modern web browsers, such as Google Chrome, Mozilla Firefox, and Microsoft Edge.

5. System Features

5.1 User account management

I Dislike It! provides basic account management features, such as creating an account and logging in. For the purpose of this application, we have opted not to use a password-based authentication system, but rather a passwordless authentication system, based on email gates. This means that the user, upon entering their email address on the sign in page, will receive an email with a code. Then, they will be prompted to enter the code in the website and we will verify that the code is correct. If it is, the user will be logged in. Furthermore, we have opted not to allow the upload of images on the website, as we believe that this would be a security risk. Since this is, in the end, a platform for expressing opinions, there is always the risk of generating negative feedback, and we do not want to encourage users to upload images that could be used to identify them or other individuals.

5.2 Categories overview

Users can browse the various categories that are available on the website and may also filter the categories by name. Additionally, users can search for categories by name.

5.3 Category overview

Registered users can view all of the posts in a certain category and, if they desire, they can press the "Add post" button to add a new post to the category. The posts are displayed in a single column, and contain the title, the author, the rating criteria and the ratings given by the author, and the description of the post. The user can click on a post to view it in more detail.

5.4 Post overview

Users can view a post in more detail. The post contains the title, the author, the rating criteria and the ratings given by the author, and the description of the post. The user can also see all the replies to the post, along with the authors and ratings of the new replies. The user may also reply to the post and express their own opinion.

5.5 Statistics

Registered users can view statistics for a category or a post. The statistics are displayed in a CSV, DocBook or PDF format. These statistics can include the average rating for a category, the average ratings for a post, the number of posts in a category, a graph of the ratings for a category, and a graph of the ratings for a post.

5.6 Notifications

Users can receive notifications by email, for the purpose of allowing authentication.

6. Other Non-Functional Requirements

6.1 Performance requirements

I Dislike It! is designed to be fast and responsive, with a page load time as short as possible. The website is lightweight and uses a minimal amount of resources, allowing it to run smoothly on a wide range of devices.

6.2 Security requirements

I Dislike It! takes user privacy and security seriously and thus, in order to avoid any risk of data breaches or leaks, we have opted not to allow the upload of images on the website. For the sign in functionality we are going to use an email based system with which the user will receive a token (that will be used as a password) or a standard password based sign in, but the password will be hashed before being stored in the database to ensure security in case of a data breach. Moreover, the communication between the frontend and the backend will be made using the HTTPS protocol, that encrypts all traffic.

6.3 Scalability requirements

I Dislike It! is designed to be scalable and can be easily adapted to accommodate a larger number of users. The website is built using a monolithic architecture, which is a good choice for a small website such as this one. However, if the website were to grow in popularity and the number of users were to increase, we could easily migrate to a microservices architecture, which would allow us to scale the website horizontally and to add new features more easily.

7.1 Design Principles

The following design principles guide the user interface design of the application:

    Simplicity: The user interface is designed to be clean, intuitive, and free from unnecessary complexity. It employs clear layouts, concise language, and intuitive navigation to enhance usability.
    Consistency: Consistency is maintained throughout the user interface design, ensuring a familiar and cohesive experience. Conventions are followed to enhance usability and reduce user confusion.

7.2 Visual Elements

Visual elements play a crucial role in the application's user interface design. The following aspects are considered:

    Colors:The application utilizes a color scheme based on blues in light mode and grays in dark mode. In light mode, shades of blue are used to create a visually pleasing and vibrant experience. In dark mode, various shades of gray are employed to provide a visually appealing and comfortable interface.
    Typography: Thoughtful typography choices ensure readability and visual hierarchy. Clear and legible fonts are used to enhance user comprehension and ease of use.
    Icons: Appropriate icons are utilized to enhance user understanding and provide visual cues. Icons are selected based on their relevance and consistency with the overall design.

7.3 Dark mode

The application features a Dark Mode option that provides a visually appealing and comfortable interface for users who prefer a darker color scheme. Dark Mode is consistent throughout the site, maintaining a unified visual experience across different pages and components.

7.3.1 Implementation and Theme Storage

The Dark Mode theme preference is stored in the user's browser using the Local Storage API. When a user enables Dark Mode, their preference is saved, allowing the application to remember and apply the Dark Mode theme consistently across subsequent visits.

7.3.2 Benefits of Dark Mode

    Reduced Eye Strain: The darker color scheme reduces the amount of bright light emitted by the screen, which can alleviate eye strain and fatigue, particularly during extended usage.
    Battery Efficiency (for OLED Displays): On devices with OLED displays, Dark Mode can potentially save battery life as individual pixels can be turned off or dimmed, resulting in lower power consumption.
    Visual Aesthetics: Dark Mode provides a sleek and modern look

8.1 Extensions

8.1.1 CategoryStatisticsCalculator

  Description
The CategoryStatisticsCalculator is a component used to generate statistics for a specific post. It performs calculations based on the provided data and produces statistical metrics that can be utilized on the frontend and for other purposes, such as generating export documents.

  Purpose
The CategoryStatisticsCalculator is designed to calculate various statistics for a specific category based on the provided data. It aims to provide insights into the category's post count, average score, ratings distribution, average score per rating, posts sorted by average score, posts and their creation dates, and posts count per day.

  Functionality

    Post Count: Calculates the total number of posts in the category.
    Average Score: Calculates the average score across all posts in the category.
    Ratings Distribution: Analyzes the distribution of ratings for each rating field in the category.
    Average Score per Rating: Computes the average score per rating field in the category.
    Posts Sorted by Average Score: Sorts the posts in the category based on their average score.
    Posts and Creation Dates: Retrieves the titles and creation dates of the posts in the category.
    Posts Count per Day: Counts the number of posts created on each day in the category

  Usage

    Frontend Display: The calculated statistics can be presented on the frontend to provide users with insights about the category's performance, ratings, and other relevant information.
    Export Functions: The statistics generated by the CategoryStatisticsCalculator can be utilized by the export functions to populate CSV, PDF and DocBook documents. These documents may include the category's post count, average score, ratings distribution, and other relevant metrics.

8.1.2 PostStatisticsCalculator

  Description
The PostStatisticsCalculator is a component used to generate statistics for a specific post. It performs calculations based on the provided data and produces statistical metrics that can be utilized on the frontend and for other purposes, such as generating export documents.

  Purpose
The purpose of the PostStatisticsCalculator is to analyze the data associated with a post and provide valuable insights through statistical calculations. It aims to offer meaningful statistics, such as the number of replies, average scores by rating, and overall average score.

  Functionality

    Replies Count: Calculates the total number of replies for the post.
    Average Scores by Rating: Computes the average scores for each rating field based on the replies.
    Overall Average Score: Calculates the average score for the post, considering all rating fields.

  Usage

Parameters: none

8.3 Architecture

8.3.1 Backend Architecture

The back-end server was built on Java, using a custom-built framework (explanations regarding the framework are below) and using a PostgreSQL database. The back-end code was structured following the Domain-Driven Design approach and on a monolithic design. As part of developing the back-end, the Test Driven Development design pattern was used, ensuring that all of the created features, on both the domain and business levels, are thoroughly tested. Furthermore, a custom Mediator was used in order to reduce code coupling (check framework docs). The backend server was also built to follow the REST paradigm and features the following resources (root level):

  • categories
  • posts
  • users
  • rss

8.3.2 Framework

For the backend server, we have developed a custom framework, specific to our needs. These are the main features of the framework:

  • HttpServer: The HttpServer is a class built upon Java’s native HttpServer implementation and customized for our specific needs. This HttpServer will be given a certain port and a list of controllers to use.

  • Controllers: Each controller class must implement the IController interface, which is an empty interface used for the sole purpose of identifying the controllers. Each controller class contains a list of methods representing endpoints, each annotated with the @Route(path = “.......”, method = HttpVerbs.[yourVerbHere]) and @Controller(name = “.....”). These custom-built annotations help define the specifics of the endpoint, such as the route, method and name.

  • Router: The Router class is responsible for intercepting the Http requests and then forwarding them to the appropriate endpoint, based on the route and http verb received. This Router implementation also permits the use of route parameters, such as in the above-described routes (i.e., posts/{id}). The route parameter will be matched to a corresponding parameter of the endpoint method.

  • Mediator: The Mediator is a feature inspired from Jimmy Bogard’s MediatR library for C#. This Mediator acts as a “hub”, facilitating the communication between the endpoint method and the Command Handler, but without directly referencing the Command Handler class in the endpoint method, to reduce code coupling. Each Command implements the IRequest interface, and that T type is to be replaced with the return type of the handler. The Command Handler class must implement the IRequestHandler interface, where T is to be replaced with the command class and V with the return type, the same as the Command. In the Main class, where all configuration happens, the commands and handlers must be declared for the Mediator and, then, when mediator.send(command) is called, the Mediator will be able to forward that command to the appropriate handler, if one exists.

  • Database and QueryProvider: The Database class is a custom implementation and acts as a point of creating connections to the database. For this, we have used Hikari, in order to use connection pooling to prevent blocking calls to the database due to the jdbc driver. The QueryProvider class acts kind of like a repository, and contains generic methods which are used to insert entities, get all entities of a certain type, or get an entity of a certain type by id.

8.3.3 Frontend Architecture

The front-end code is structured in a modular design, split into services, looks, models, errors, routes and components. Each page accessed represents a Component (i.e: PostOverviewComponent is the page where you view a post and its replies). Furthermore, since each component needs to call at least one endpoint, we have created a number of services in the shared component, in order to reduce code duplication and decrease the package size and increase performance. A short summary of the responsibilities of each “module”:
  • components: the components represent pages and their custom build javascript code, along with the html and css files.

  • services: services are javascript classes which, each, represent one of the controller classes in the backend (authentication, categories, posts, statistics, users). Each of these services contains methods that make a call to the backend server and return the response either in the form of an Error, or in the form of a model.

  • models: the models are javascript classes which represent our Domain objects (i.e: post, user, etc…) and act as containers for data, to maintain a more organized code.

  • errors: the errors class is a collection of all of the possible errors which can be returned from the backend and is used to display custom messages for each error. In case an unexpected error comes from the backend, an appropriate message will be displayed.

  • look: the look module contains the “behind the scenes” work for the dark mode implementation and for certain pop-ups.

  • routes: the routes class is a collection of all of the routes available in the backend and has the possibility of switching the prefix (from localhost to the host which hosts our backend server), to accommodate running either on localhost or on the web.

8.3.4 Docker and Deployment

As part of this project, we have also created Dockerfiles in order to run our backend and frontend apps and our database in Docker containers. The backend server runs on an Ubuntu container, and the frontend container runs with the help of an Nginx server, and the database is a separate PostgreSQL container. With these dockerfiles, we also had the possibility of deploying the web-app on a dedicated host in a very simple manner. For deployment we have chosen Northflank as our provider, as it offered a free tier for small apps, and it was relatively easy to configure.
